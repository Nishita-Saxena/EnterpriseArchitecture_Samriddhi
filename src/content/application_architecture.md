# Application Architecture Overview

The Samriddhi application architecture follows a **microservices-based, API-first design** that enables independent deployment, scalability, and operational resilience. Each service owns a specific business capability and communicates with other services through secured APIs and event-driven interactions.

The architecture is organized into three logical layers:

* Application Interface Layer
* Application Services Layer
* Application Components Layer

Shared platform capabilities such as identity management, event streaming, notifications, and API management provide consistency, security, and interoperability across the ecosystem.

---

## Architectural Principles

The application architecture is governed by the following principles:

### Microservices First

Business capabilities are implemented as independently deployable services with clearly defined responsibilities.

### API-Driven Communication

Service interactions occur exclusively through published APIs and event streams. Direct database access between services is prohibited.

### Event-Driven Integration

Business events are exchanged through a centralized event backbone, enabling asynchronous collaboration and reducing runtime dependencies.

### Security by Design

Authentication, authorization, and identity management are centralized through NCRID and the API Gateway.

### Resilience and Scalability

Services can scale independently, and failure of a single service does not impact the overall platform.

---

## Application Interface Layer

The Application Interface Layer provides role-specific access to platform capabilities.

Primary interfaces include:

* Resident Mobile Application
* RWA Committee Dashboard
* Vendor Mobile Application
* Municipal Officer Portal
* Insurance Provider Portal
* Public APIs

Residents use the platform for incident reporting, status tracking, notifications, and community engagement. Administrative users access operational dashboards, compliance monitoring, vendor management, and analytics. External stakeholders interact through dedicated portals and secured APIs.

---

## Application Services Layer

The Application Services Layer delivers the core business capabilities of the platform.

| Service                     | Primary Responsibility                                                                      |
| --------------------------- | ------------------------------------------------------------------------------------------- |
| Incident Management Service | Incident lifecycle management, SLA tracking, escalation, closure, and evidence coordination |
| Vendor Marketplace Service  | Vendor onboarding, bidding, work allocation, SLA monitoring, and performance management     |
| Compliance Tracker Service  | Compliance scheduling, reminders, documentation, and audit support                          |
| IoT Intelligence Service    | Sensor processing, anomaly detection, predictive alerts, and maintenance insights           |
| Blockchain Evidence Service | Immutable evidence management and verification                                              |
| NCRID Identity Service      | Authentication, authorization, consent, and identity management                             |
| Notification Service        | Push, SMS, email, and WhatsApp communications                                               |
| API Gateway Service         | Routing, authentication, rate limiting, versioning, logging, and security enforcement       |

---

## Event-Driven Collaboration Model

Application services collaborate through an event-driven architecture implemented using Apache Kafka.

Representative business events include:

* INCIDENT_CREATED
* VENDOR_ASSIGNED
* SLA_BREACHED
* PREDICTIVE_ALERT
* INCIDENT_CLOSED
* NOTIFICATION_REQUESTED

This model enables asynchronous communication, improves resilience, and minimizes direct service dependencies.

For example, incident events generated by the Incident Management Service can trigger notifications, while predictive alerts generated by the IoT Intelligence Service can automatically initiate incident workflows.

---

## Application Components

Business services are implemented through dedicated software components:

* Incident Management Component
* Vendor Marketplace Component
* Compliance Tracker Component
* IoT Intelligence Component
* Blockchain Registry Component
* NCRID Component
* Notification Component
* API Gateway Component

Each component encapsulates its own business logic, deployment lifecycle, and operational responsibilities, enabling independent development, testing, deployment, and scaling.

---

## Security and Access Management

Security controls are consistently applied across all application services.

Key controls include:

* OAuth 2.0 and OpenID Connect authentication
* JWT-based authorization
* Role-based access control (RBAC)
* Multi-factor authentication for privileged users
* TLS-secured communications
* API Gateway policy enforcement
* Consent-driven identity management through NCRID

These controls ensure that users access only the information and capabilities relevant to their assigned roles and responsibilities.

---

## Application Collaboration Summary

The application architecture establishes a modular platform that integrates incident management, vendor coordination, compliance monitoring, predictive maintenance, identity management, and stakeholder communication within a unified ecosystem.

Through microservices, API-first communication, event-driven integration, and federated identity management, the architecture provides a scalable, secure, and resilient foundation for large-scale deployment across housing societies.
