# Component Architecture

## Purpose
This document defines the reusable component architecture for the Samriddhi architecture review portal. Components must support markdown-driven pages, hierarchy-aware navigation, and diagram-first content presentation.

## Layout Components
### MainLayout
Responsibilities:
- Provide the global HTML shell and page metadata.
- Establish the desktop-first portal frame.
- Host sidebar, header, and content region placement.

Relationships:
- Wraps every page and document view.
- Composes `Header`, `Sidebar`, and the page content slot.

### Header
Responsibilities:
- Display the portal title and current section context.
- Provide compact utility actions where needed.
- Reinforce that the product is a review portal, not a marketing site.

Relationships:
- Sits inside `MainLayout` above the content area.

### Sidebar
Responsibilities:
- Render the hierarchical navigation tree.
- Support collapse and expand behaviour.
- Preserve the architecture section ordering.

Relationships:
- Controlled by `MainLayout`.
- Uses `SidebarNavigation` for structure.

## Navigation Components
### SidebarNavigation
Responsibilities:
- Render top-level sections and nested pages.
- Highlight the active route.
- Support section grouping and linear document traversal.

Relationships:
- Used by `Sidebar`.
- Feeds `Breadcrumbs` and `PreviousNextNavigation` with location data.

### Breadcrumbs
Responsibilities:
- Show the current document path in the hierarchy.
- Help reviewers orient within the portal.

Relationships:
- Used within document pages and the header area where appropriate.

### PreviousNextNavigation
Responsibilities:
- Provide ordered navigation between adjacent pages.
- Support review workflows that move through architecture artifacts sequentially.

Relationships:
- Consumes the site order map defined by the content metadata.

## Architecture Content Components
### ArchitecturePage
Responsibilities:
- Assemble the title, metadata bar, diagram, content, and navigation.
- Enforce the standard document page layout.

Relationships:
- Uses `MetadataBar`, `DiagramViewer`, `MarkdownRenderer`, and `PreviousNextNavigation`.

### MetadataBar
Responsibilities:
- Display document title, category, version, and status.
- Provide consistent metadata treatment across all pages.

Relationships:
- Rendered at the top of every architecture document page.

### DiagramViewer
Responsibilities:
- Render the associated architecture diagram when available.
- Support zooming and fullscreen inspection.

Relationships:
- Sits directly above markdown content on diagram-backed pages.

### MarkdownRenderer
Responsibilities:
- Render approved markdown exactly as authored.
- Avoid summarisation, rewriting, or interpretation.

Relationships:
- Consumes content sourced from the Astro content collection.

## Shared UI Components
### Card
Responsibilities:
- Hold grouped information with a consistent visual container.

### Badge
Responsibilities:
- Present short metadata labels such as status or version.

### Section
Responsibilities:
- Structure related content into readable groups.

### Divider
Responsibilities:
- Separate content areas without visually overwhelming the page.

## Component Relationships
- `MainLayout` is the root shell.
- `Sidebar` and `Header` provide global navigation and orientation.
- `ArchitecturePage` is the canonical document page composition.
- `MarkdownRenderer` and `DiagramViewer` are content-bearing leaf components.
- Shared components provide consistent styling primitives across the portal.
- Navigation state should be derived from the content collection metadata so ordering and routing remain aligned.

---------------

## Additional Architecture Components

### TableOfContents

Responsibilities:

* Generate navigation automatically from markdown headings.
* Provide quick access to sections within long architecture documents.
* Highlight the currently active section while scrolling.
* Improve review efficiency for architecture stakeholders.

Relationships:

* Consumes headings generated by `MarkdownRenderer`.
* Displayed within `ArchitecturePage`.
* Synchronizes with document scroll position.

---

### DiagramGallery

Responsibilities:

* Provide a centralized view of all architecture diagrams.
* Allow reviewers to browse architecture artifacts visually.
* Support navigation from diagrams to associated architecture documents.
* Improve discoverability of architecture deliverables.

Relationships:

* Consumes architecture image metadata.
* Links directly to corresponding `ArchitecturePage` routes.
* Uses `DiagramViewer` for rendering and interaction.

---

### SearchBar

Responsibilities:

* Search architecture documents, page titles, and metadata.
* Support quick navigation to architecture artifacts.
* Improve accessibility of architecture knowledge across the portal.
* Enable reviewers to locate specific architecture concepts efficiently.

Relationships:

* Available globally through `Header`.
* Uses content collection metadata and markdown content.
* Provides navigation links to matching architecture pages and sections.

---

## Extended Component Relationships

The architecture portal should support three primary review workflows:

### Document Review Workflow

Reviewers navigate architecture documents sequentially using:

* SidebarNavigation
* Breadcrumbs
* PreviousNextNavigation
* TableOfContents

---

### Diagram Review Workflow

Reviewers navigate architecture diagrams using:

* DiagramGallery
* DiagramViewer
* ArchitecturePage

This workflow supports architecture-focused reviews where diagrams are the primary artifacts.

---

### Knowledge Discovery Workflow

Reviewers locate information using:

* SearchBar
* SidebarNavigation
* Breadcrumbs

This workflow supports rapid discovery of architecture decisions, principles, risks, governance controls, and implementation details.

---

## Component Priority

Core Components:

* MainLayout
* Sidebar
* Header
* ArchitecturePage
* MarkdownRenderer
* DiagramViewer

Supporting Components:

* MetadataBar
* Breadcrumbs
* PreviousNextNavigation
* TableOfContents

Enhancement Components:

* SearchBar
* DiagramGallery

Core components are mandatory for initial implementation.

Supporting and enhancement components should be designed as reusable extensions without altering the architecture content rendering pipeline.
